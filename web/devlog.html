关键问答
Q
卡片黄色和白色是什么意思？
A：relevance_score ≥ 0.8 → 黄色背景，表示 Claude 评为重要且多源验证。<0.8 → 白色普通。重设计后将改为信源角标+分数徽章。
Q
股票图表是日线还是月线？
A：yfinance period="30d"，拉约21个交易日日线收盘价，迷你折线图展示近一个月走势。
Q
Ticker 1.00 是什么单位？
A：relevance_score 0.0→1.0 满分指数，Claude 打基础分后叠加多信源加权（+0.08/源，上限1.0）。Ticker 已改为显示信源名，不显示数字。
Q
刷新按钮的作用？
A：重新从 data/*.json 读取文件并重绘页面（cache:'no-store'）。不触发 Python 管线，不联网抓取。用于 Pages 已重新部署后手动拿最新数据。日常走 launchd 自动，不需要手动点。
Q
Ticker 显示哪些 Tab 的内容？
A：全部三个 Tab（近期要闻 + 模型&榜单 + 投融资&二级）的高分条目都会进 Ticker，不按分类过滤，只按 score ≥ 0.88 过滤。

信源评级方式

信源按层级（Layer）和主观质量星级双维度评估。层级影响 Claude 打分时的信息密度预期，星级用于未来接入优先级判断。

层级 特征 代表信源 对打分的影响
扫描层 高频，广覆盖，编辑筛选 Techmeme, 量子位, 36氪, VentureBeat 条目多→基础噪音源，多源重叠时权重叠加
社区层 实时热度，工程师视角 HN, Reddit ML 与扫描层重叠时强力拉高评分（工程师+媒体=双重验证）
通讯层 低频，深度，作者权威 Import AI, SemiAnalysis, Latent Space 出现即倾向高分，Claude 对深度分析稿件给分更高
研究层 学术前沿，批量预印本 arXiv AI 多数低分，被媒体/社区同步报道时分数大幅拉高

星级定义（主观，基于信噪比）：★★★★★ 极高信噪比，每篇产出价值大 / ★★★★ 高质量，少量噪音 / ★★★ 中等，需依赖过滤 / ★★ 噪音较多

新闻评分方式

执行者：claude -p（调用本地 Claude Code CLI，复用付费订阅），每日运行一次，处理约 180 条原始条目。

输入格式：[ID|来源] 标题（摘要前100字）

Claude 处理步骤：
  Step 1 → 跨信源归组：同一事件的多信源报道合并为一组
  Step 2 → 每组输出：
              relevance_score  0.0–1.0（基础评分，不含多源加权）
              category         模型发布 / 融资并购 / 产品动态 /
                               研究技术 / 行业动态 / 算力供应链 / 社区信号
              summary_cn       1–2句中文摘要，直接说事
  Step 3 → 程序加权（不经过 Claude）：
              +0.08 per extra source（上限 1.0）
              最终 relevance_score = min(1.0, base + (sources-1) * 0.08)

阈值说明：
  ≥ 0.88  → 自动写入 pinned.json（Ticker 置顶，保留14天）
  ≥ 0.80  → 前端显示黄色卡片（高相关标记）
  ≥ 0.50  → 中等相关，正常展示
  < 0.50  → 低相关，展示但不优先（多为非AI内容）

关注范围（高分区）：AI 模型/Agent/产品/融资/算力/行业应用。
低分区（不删除）：与 AI 无关的科技新闻、政治事件、老技术话题。

注意：评分是 Claude 的主观判断，同一事件在不同上下文中可能得分不同。多信源加权（+0.08/源）是程序硬规则，比 Claude 主观判断更稳定可靠。满分 1.0 通常意味着：Claude 认为极度相关 + 至少3个独立信源同时报道。
抓取与刷新机制
触发机制（三种）：
  1. 自动   launchd 10:00 每日（~/Library/LaunchAgents/com.techradar.daily.plist）
            Mac 睡眠时触发时间会顺延至下次唤醒后立即执行
  2. 手动   bash schedule/run_daily.sh（从项目根目录）
  3. 部分   cd pipeline && .venv/bin/python pull_rss.py（单步调试）

执行顺序：
  1. pull_openrouter.py    → OpenRouter API（JSON，实时）
  2. pull_rss.py          → 29个 RSS/Atom feed（feedparser）含 P1 企业 + P3 分析
  3. pull_stocks.py       → yfinance（美股 + A股 + 港股）
  4. pull_github_trending.py → GitHub trending HTML（BeautifulSoup）
  5. pull_huggingface.py  → HuggingFace API（sort=likes7d）
  6. pull_filings.py      → SEC EDGAR API（8支美股 10-K/10-Q 直链）[P2]
  7. pull_score.py        → claude -p 打分归组（超时 600s）
                          → 更新 archive.json（30天滚动）
                          → 更新 pinned.json（score≥0.88 置顶14天）
  8. git add data/ && git commit && git push
     → 触发 GitHub Actions → Pages 重新部署（约2分钟）

时效过滤：
  - pull_rss.py 阶段：每条 entry 检查 published_parsed，> 7天跳过
  - pull_score.py 阶段：读入时二次过滤 published_at，> 7天跳过
  - 两层过滤确保即使个别 RSS 返回历史存档也不会进入 digest

超时设置：
  - 单个 RSS fetch：20s（含重试3次，指数退避）
  - pull_score.py claude -p：600s（约180条条目，实测约5分钟）
  - pull_stocks：每只股票独立请求，无全局超时

前端刷新（浏览器端）：
  - 点击"↻ 刷新数据"：重新 fetch data/*.json（cache:'no-store'）
  - 不触发管线，只读取已有 JSON 文件
  - GitHub Pages 有 CDN 缓存，新数据推送后约1–3分钟生效

• 新智元（xinzhiyuan.com）：域名解析到 198.18.1.253（IANA 保留地址），网站实际不可访问，无 RSS。
